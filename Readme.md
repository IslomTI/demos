# 🏆 Демоэкзамен 2025: Автоматизация Модулей 1 и 2

![Bash](https://img.shields.io/badge/Language-Bash-4EAA25?style=for-the-badge&logo=gnu-bash&logoColor=white)
![Ubuntu](https://img.shields.io/badge/OS-Ubuntu_Linux-E95420?style=for-the-badge&logo=ubuntu&logoColor=white)
![Специальность](https://img.shields.io/badge/Специальность-09.02.06-blue?style=for-the-badge)

Отказоустойчивый комплекс shell-скриптов для безошибочного развертывания инфраструктуры в рамках промежуточной аттестации и регионального чемпионата (Сетевое и системное администрирование). 

Архитектура разделена на два независимых скрипта (Модуль 1 — Сеть, Модуль 2 — Сервисы), чтобы сбой при скачивании тяжелых пакетов не затронул фундаментальную маршрутизацию и гарантировал сохранение баллов за сетевую топологию.

🔗 **Репозиторий содержит:**
* `iti7.1.sh` — Базовая сеть, Виртуальный коммутатор (Bridge + PVID через carrier.d), GRE-туннели, OSPF, DHCP, Double NAT, DNS (Bind9).
* `iti7.2.sh` — Nginx Reverse Proxy, LVM-Safe RAID0 (Mdadm), NFS + AutoFS, Samba AD DC, Docker (MediaWiki), Apache (Moodle 4.0.4).

## ⚠️ ЭТАП 1: Аппаратная подготовка ВМ (КРИТИЧНО)

Если не выполнить эти требования **до** запуска виртуальных машин, установка сервисов зависнет, или скрипт выдаст фатальную ошибку при сборке RAID-массива.

### 1. Распределение ОЗУ (RAM)
* **`BR-SRV`** — Минимум **3 ГБ** (рекомендуется 4 ГБ). Это самый нагруженный узел (Samba AD DC + Docker).
* **`HQ-SRV`** — Минимум **2 ГБ** (MariaDB + Moodle + Bind9).
* **`ISP`, `HQ-RTR`, `BR-RTR`, `HQ-CLI`** — Достаточно по **1 ГБ**.

### 2. Подготовка дисков для RAID0 (Только для HQ-SRV)
В настройках виртуальной машины **HQ-SRV** необходимо вручную добавить жесткие диски:
1. Нажмите `Edit virtual machine settings` ➔ `Add...` ➔ `Hard Disk`.
2. Выберите тип **SCSI** или **SATA** (Обязательно!).
3. Добавьте **ТРИ новых пустых диска** (например, по 5 ГБ каждый).
> 💡 *Скрипт 2-го модуля автоматически просканирует шину, найдет эти 3 пустых диска (отфильтровав системный LVM-том) и аппаратно соберет из них массив `/dev/md0` для расшаривания по NFS.*

## 🌐 ЭТАП 2: Топология и Сетевые адаптеры (LAN Segments)

Для исключения бродкаст-штормов и конфликтов с хостовой ОС, топология строится **строго через LAN Segments** (а не через Host-only VMnet). 
В настройках любой ВМ перейдите в `Network Adapter` ➔ `LAN Segments...` ➔ `Add` и создайте 5 изолированных сегментов:

* **`ISP-HQ`** — Магистраль `ISP` ⟷ `HQ-RTR`
* **`ISP-BR`** — Магистраль `ISP` ⟷ `BR-RTR`
* **`HQ-LAN-SRV`** — Локальная сеть `HQ-SRV` (Access VLAN 10)
* **`HQ-LAN-CLI`** — Локальная сеть `HQ-CLI` (Access VLAN 20)
* **`BR-LAN`** — Локальная сеть Филиала

### Схема подключения адаптеров в настройках ВМ:

| Имя машины | Адаптер 1 (Временный) | Адаптер 2 (LAN Segment) | Адаптер 3 (LAN Segment) | Адаптер 4 (LAN Segment)|
| :--- | :--- | :--- | :--- | :--- |
| 🌍 **ISP** | `NAT` (Интернет) | `ISP-HQ` | `ISP-BR` | - |
| 🖧 **HQ-RTR** | `NAT` | `ISP-HQ` | `HQ-LAN-SRV` | `HQ-LAN-CLI` |
| 🖧 **BR-RTR** | `NAT` | `ISP-BR` | `BR-LAN` | - |
| 🖥️ **HQ-SRV** | `NAT` | `HQ-LAN-SRV` | - | - |
| 🖥️ **BR-SRV** | `NAT` | `BR-LAN` | - | - |
| 💻 **HQ-CLI** | `NAT` | `HQ-LAN-CLI` | - | - |

## 🚀 ЭТАП 3: Инструкция по запуску (Два прохода)

Механика работы скрипта полагается на строгий порядок запуска. Сначала мы выстраиваем L2/L3 маршрутизацию (Pass 1), затем разворачиваем прикладное ПО L7 (Pass 2).

### Шаг 1. Скачивание скриптов
Включите все машины. Убедитесь, что интернет работает (через первый `NAT` адаптер). Зайдите на каждую ВМ под пользователем `root` (`sudo su`) и выполните:
```bash
wget [https://raw.githubusercontent.com/IslomTI/demos/main/iti7.1.sh](https://raw.githubusercontent.com/IslomTI/demos/main/iti7.1.sh)
wget [https://raw.githubusercontent.com/IslomTI/demos/main/iti7.2.sh](https://raw.githubusercontent.com/IslomTI/demos/main/iti7.2.sh)
chmod +x iti7.1.sh iti7.2.sh

```

### Шаг 2. ИЗОЛЯЦИЯ СЕТЕЙ (Ловушка адаптеров)

Не выключая машины, перейдите в их настройки в VMware (`VM -> Settings`).
У **ВСЕХ машин КРОМЕ ISP** выберите первый адаптер (`NAT`) и **СНИМИТЕ галочку "Connected"**.

> 🛑 **Внимание при вводе данных:** Отключенный NAT-интерфейс (обычно `ens33`) всё ещё существует в системе, но он аппаратно обесточен. Когда скрипт попросит ввести имя интерфейса, указывайте **рабочие** адаптеры (например, `ens34`, `ens35`), смотрящие в нужные LAN Segments. Если ввести отключенный интерфейс, роутер останется без связи!

### Шаг 3. Запуск по цепочке

**Проход 1 (Заливаем сетевой фундамент — Модуль 1):**

1. Сначала: **ISP** (`./iti7.1.sh` ➔ пункт `1`).
2. Затем роутеры: **HQ-RTR** (пункт `2`) и **BR-RTR** (пункт `3`).
3. Затем серверы: **HQ-SRV** (пункт `4`) и **BR-SRV** (пункт `5`).
4. Клиент: **HQ-CLI** (пункт `6`). *(Скрипт использует нативный DNS-поллинг и будет ждать получения IP по DHCP).*

*(На этом этапе сеть полностью маршрутизируется, OSPF-соседство установлено, а DNS-имена резолвятся. Настоятельно рекомендуется сделать Snapshot всех виртуалок).*

**Проход 2 (Поднимаем сервисы — Модуль 2):**

Запускать можно параллельно на разных машинах:

1. **ISP** (`./iti7.2.sh` ➔ пункт `1` — Настроит Nginx Reverse Proxy для обхода Double NAT).
2. **HQ-SRV** (`./iti7.2.sh` ➔ пункт `4` — Сборка RAID0, NFS, MariaDB, Apache, Moodle).
3. **BR-SRV** (`./iti7.2.sh` ➔ пункт `5` — Samba AD DC, TCP-поллинг DNS, запуск MediaWiki в Docker).
4. **HQ-CLI** (`./iti7.2.sh` ➔ пункт `6` — Ввод в домен, настройка SSSD без FQDN, AutoFS).

* *После настройки клиента обязательно выполните команду `reboot`, чтобы применились групповые политики домена (mkhomedir).*

## 📸 Чек-лист для отчета экспертам

После завершения настройки пройдитесь по машинам и сделайте скриншоты для подтверждения работоспособности.

### 🛡️ Модуль 1: Сетевая инфраструктура

**1. Проверка выдачи адресов (DHCP)**
На клиенте **HQ-CLI** выполните:

```bash
ip a

```

*Ожидаемый результат: Наличие IP-адреса из пула 192.168.20.10-30.*

**2. Проверка динамической маршрутизации (OSPF) и GRE**
На роутере **HQ-RTR** выполните:

```bash
vtysh -c "show ip ospf neighbor"

```

*Ожидаемый результат: Виден сосед `2.2.2.2` в состоянии `Full`.*

С клиента **HQ-CLI** пропингуйте сервер филиала (проверка туннеля и моста):

```bash
ping 192.168.30.2 -c 4

```

**3. Проверка DNS (Прямая, обратная зона и пересылка)**
На клиенте **HQ-CLI** выполните:

```bash
host hq-rtr.au-team.irpo
host 192.168.10.1

```

**4. Проверка безопасного удаленного доступа и DNAT**
На клиенте **HQ-CLI** подключитесь к серверу по нестандартному порту:

```bash
ssh sshuser@192.168.10.2 -p 3015

```

*Ожидаемый результат: Появление баннера "Authorized access only". Пароль для входа: `P@ssw0rd`.*

### ⚙️ Модуль 2: Сервисы

**1. Проверка аппаратного RAID0**
На сервере **HQ-SRV** выполните:

```bash
lsblk
mdadm --detail /dev/md0

```

*Ожидаемый результат: Видно 3 диска, объединенных в активный массив `md0`.*

**2. Проверка контроллера домена Samba AD DC**
На сервере **BR-SRV** выполните:

```bash
samba-tool domain info AU-TEAM.IRPO
samba-tool user list

```

*Ожидаемый результат: Статус домена и список созданных пользователей (user1..5.hq).*

**3. Проверка синхронизации времени (NTP)**
На сервере **HQ-SRV** или **BR-SRV** выполните:

```bash
chronyc sources

```

*Ожидаемый результат: Звездочка `*` слева от IP-адреса локального роутера, доказывающая успешную синхронизацию.*

**4. Проверка AutoFS и входа в домен (SSSD)**
На клиенте **HQ-CLI** в графическом интерфейсе войдите под коротким логином `user1.hq` с паролем `P@ssw0rd123!`. Откройте терминал:

```bash
id
df -h | grep nfs

```

*Ожидаемый результат: Команда `id` покажет доменные группы, а `df -h` — автоматически примонтированную папку `raid0`.*

**5. Проверка Web-сервисов (Nginx Reverse Proxy + Docker)**
В графическом интерфейсе **HQ-CLI** откройте Яндекс Браузер и сделайте скриншоты:

1. Адрес `http://moodle.au-team.irpo` *(Откроется Moodle).*
2. Адрес `http://wiki.au-team.irpo` *(Откроется MediaWiki).*
